── phase 1 ──

VIBE CODING

EASY MODE
  • just describe what you want
  • fast results, felt like magic
  • great for small projects
  • low friction to start
GAME OVER
  • codebase grew, AI lost the thread
  • fix one, break two others
  • context drifted and tangled
  • fighting the tool, not building
"vibe coding is great until
you're not vibing anymore"
── phase 2 ──

SPEC-DRIVEN DEV

01
write the spec first
define what you're building before writing a single prompt
02
give the AI a contract, not a wish
vague prompt → vague result
clear spec → consistent output
03
results got dramatically better
codebase stayed manageable
more building, less untangling

── honest assessment ──

WHERE AI GOT STUCK

✗
on-screen joystick
several prompts, several iterations. never got it working reliably.
✗
npc interaction range
triggered walking past npcs — too far away, no stopping required. kept breaking after fixes.
✗
ai-generated sprites
tried many times to get good otter art from ai image tools. never got output we liked.
── game over · lessons learned ──

TAKEAWAYS

★
start with something purely for fun
no stakes = faster learning. build a thing you want to exist.
★
when vibe coding stalls, write a spec
clear intent → better output. this is just good engineering.
★
the barrier is lower than you think
a parking lot. a phone. an html file. that's enough to start.
