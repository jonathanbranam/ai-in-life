Phase 1

Vibe Coding:
It Worked, Until It Didn't

The Fun Part

  • Low friction — just describe what you want
  • Fast results on small projects
  • Felt like magic

The Wall

  • Codebase grew, AI lost the thread
  • Fix one thing, break two others
  • Fighting the tool instead of building
"Vibe coding is great until you're not vibing anymore."
Phase 2

Spec-Driven Development

Write the spec first

Define what you're building, how it should behave, what the constraints are — before writing a single prompt.

Give the AI a contract, not a wish

A vague prompt gets a vague result. A structured spec gives the model something to hold itself accountable to.

Results got dramatically better

The codebase stayed manageable. I spent more time building, less time untangling.

The best part

Built in a Parking Lot,
on My Phone

My son was running video at a basketball game. I was waiting in the parking lot.

I opened Claude on mobile and started describing the game I wanted to build.

Claude would output an HTML file. I'd open it, play it, send feedback, repeat.

No dev environment. No laptop. Just a parking lot and a few minutes between plays.

The payoff

Life of an Otter 🦦

[ OtterGame screenshot ]

A casual game built with my daughter about an otter's daily life.

She had opinions. The otter needed to eat fish. It needed to float on its back.

The AI let us focus on the fun parts — what should the otter do? — instead of the plumbing.

What I'd tell you to try

Takeaways

Start with something purely for fun

No stakes = low pressure = faster learning. Build a thing you actually want to exist.

When vibe coding stalls, write a spec

The more clearly you define what you want, the better the AI can help you build it.

The barrier is lower than you think

A parking lot. A phone. An HTML file. That's enough to start.
